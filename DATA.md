# Data

This document describes the current data pipeline as implemented in the codebase.

The hackathon brief mentions UK sales history, monthly budget or target estimate, and a promotion plan. In the source Excel available to this project, we have reliable sales history and a promotion plan. We do not have a clean official budget table, so the app creates a derived `target_hl` estimate and labels it as target/budget in the UI.

## Source Files

Expected local files:

```text
backend/app/data/raw/
├── UK DATA.xlsx
└── Damm Trade Plan - promotions.xlsx
```

These files are confidential and must not be committed.

### `UK DATA.xlsx`

Used sheets:

| Sheet | Use |
|---|---|
| `DATABASE` | Monthly sales rows, including `Hl`, `Venta Neta`, `Margen Bruto`, period, customer, material |
| `MaterialData` | SKU metadata: brand, pack, alcohol, litres per SKU |
| `CUSTOMERS` | UK customer metadata and channel/sub-channel |

Important raw columns:

| Raw column | Current use |
|---|---|
| `AÑO CALENDARIO` | Parsed to month-start `date` |
| `Cod. Cliente` | Extract numeric customer id for join |
| `Cod. Material` | Extract material/SKU id |
| `Hl` | Hectolitres sold; this is the main model target |
| `Venta Neta` | Net revenue; used for price per hL and GBP gap approximations |
| `Margen Bruto` | Margin input for analysis/ROI context |
| `SubChannel` | Main app "channel" dimension |
| `Sales Channel` | Coarser channel, used as metadata/model feature |

### `Damm Trade Plan - promotions.xlsx`

Used as a weekly grocery promotion calendar. It has one sheet per retailer with different layouts, so `etl.py` has retailer-specific parsers.

The output is normalized into `promos.parquet` with:

- anonymized grocer label
- SKU label as written in the promo file
- ISO week
- price or multi-buy offer
- promo type
- `on_promo`
- baseline price estimate
- raw cell value

## ETL

Run:

```bash
make data
```

Main code:

- `backend/app/services/etl.py`
- `backend/app/services/anonymize.py`
- `backend/app/services/external.py`

Pipeline:

```text
DATABASE
  -> parse periods
  -> extract customer/material ids
  -> remove null-Hl accounting rows
  -> net returns/credits by customer x material x month
  -> join UK customers
  -> join material metadata
  -> aggregate to material_id x sub_channel x month
  -> add external monthly signals
  -> write wide_monthly.parquet

Promotions workbook
  -> parse each retailer sheet
  -> classify promo cells
  -> anonymize retailer names
  -> write promos.parquet

Targets
  -> derive target_hl from prior-year actuals / trailing median
  -> write targets.parquet
```

## Hectolitres

The model predicts monthly `Hl`: hectolitres sold.

`wide_monthly.parquet` is the primary table for this:

```text
material_id + sub_channel + date -> Hl
```

So a forecast point means:

```text
SKU + channel + future month -> predicted hectolitres
```

The app also estimates GBP impact by converting hL gap into money using historical `Venta Neta / Hl`.

## Targets / Budget

There is no reliable official budget table currently consumed by the app.

`targets.parquet` is generated by code:

```text
target_hl = prior_year_same_month_actual
fallback = trailing 3-month median
```

This satisfies the brief's "budget or target estimate" wording, but it should be explained as a derived target, not a Damm-provided budget.

## Negative / Null Values

Current handling:

- Null `Hl` rows are treated as accounting noise, not forecastable demand.
- Negative `Hl` rows are returns or credit-note effects.
- The ETL nets them at monthly customer/material level, then filters to positive net volume for training.

This is why raw Excel totals and model-training totals may differ.

## Event importance and evidence for the boost

UK beer demand reacts to one-off mega-events (football tournaments, Christmas). Our forecast model can't legitimately learn this from history because the events fire every 2–4 years — with a 3-year training window, LGB sees 0–1 instances and overfits. Instead, the ensemble step ([services/forecast/ensemble.py](backend/app/services/forecast/ensemble.py)) applies a small deterministic `+5%` boost to the baseline forecast for months containing a one-off mega-event (`World Cup`, `Euros`). Recurring events (Christmas, Wimbledon, bank holidays) are **not** boosted here because the model already captures them through seasonal features (`month`, `is_christmas_month`, `is_summer`, `uk_holidays_count`).

The `+5%` figure is calibrated, not arbitrary:

**Measured from our own `wide_monthly.parquet`** (all-brand UK retail, Jun+Jul vs Apr+May ratio):

| Year | Jun+Jul / Apr+May | Notes |
|---|---|---|
| 2023 | 1.279 | No major sport (baseline) |
| 2024 | 1.011 | **Euros 2024** |
| 2025 | 0.982 | Wimbledon only |

Detrended Euros lift (2024 ratio / 2025 ratio): **≈ +3%** at the monthly category level.

**Cross-check vs industry literature**: WSTA / The Grocer commonly cite **5–15% beer lift during major tournament match days**, dropping to ≈3–7% at the monthly aggregate because most of the lift concentrates on match-day weekends rather than spreading evenly across the month.

We picked **+5%** as a calibrated midpoint — above our measured 2024 Euros baseline but conservative against the wider literature band. Implementation: `ONEOFF_BOOST = 0.05` in [services/calendar.py](backend/app/services/calendar.py).

Per-event importance levels (`high` / `medium` / `low`) live in the same file and drive a separate multiplier used by the *simulator's promo lift* — see [MODEL.md](MODEL.md) for that flow.
